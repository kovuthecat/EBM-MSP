## T-003 — Moteur : évaluateur de conditions + sélection d'options

### Objectif

Écrire un module TS pur qui, pour un nœud + un objet de critères, renvoie les options **applicables**,
ordonnées, avec la liste des conditions satisfaites (le « pourquoi »), sans score caché.

### Décision clé

Sémantique des `conditions` (d'après le brief §11) :

- Une option est **applicable** si **toutes** les chaînes de son tableau `conditions` sont vraies.
- Une chaîne peut contenir des comparaisons (`==`, `!=`, `<`, `<=`, `>`, `>=`) sur une variable de
  critère et une valeur (nombre, booléen `true/false`, ou libellé d'enum), combinées par `AND` / `OR`
  (`AND` prioritaire sur `OR` ; pas de parenthèses nécessaires pour le nœud A — supporter au moins
  `OR` de comparaisons simples, et l'`AND`).
- L'option dont `conditions == ["default"]` est la **repli** : applicable seulement si aucune option
  non-default ne l'est.
- **Ordre** : préserver l'ordre des options du nœud ; l'auteur du contenu ordonne du plus spécifique au
  plus général (default en dernier). L'UI (S4) prendra la 1re applicable comme « recommandée » + les autres
  en alternatives.

### Lire / Modifier

- Créer : `src/features/decision/engine/conditions.ts` (parse + évaluation d'une chaîne),
  `src/features/decision/engine/evaluateNode.ts` (`evaluateNode(node, criteria) → { applicable: Option[],
  reasons: Map<optionId, string[]> }`), `src/features/decision/engine/*.test.ts`.

### Étapes

1. `conditions.ts` : tokenizer minimal → évaluer une comparaison atomique contre `criteria[nom]`
   (typer : nombre vs enum/string vs booléen) ; composer `AND`/`OR`. Rejeter proprement une variable
   inconnue (erreur explicite, testée) plutôt que de la traiter comme fausse silencieusement.
2. `evaluateNode.ts` : appliquer la sémantique ci-dessus (toutes les chaînes ANDées ; gestion `default` ;
   ordre conservé). Retourner aussi les conditions satisfaites par option pour alimenter le « pourquoi ».
3. Tests Vitest sur le **nœud cible réel** couvrant les 4 bandes du brief §11 :
   - jeune (<60), diabète récent (<10 ans), risque hypo faible, espérance longue, non fragile → **cible stricte** ;
   - profil intermédiaire → **default (~7 %)** ;
   - ≥ 75 ans **ou** fragile **ou** risque hypo élevé **ou** espérance limitée → **cible souple** ;
   - fragile **et** espérance limitée → **cible la moins contraignante** ;
   - plus : variable inconnue → erreur ; option `default` non retenue si une autre s'applique.

### Validation

- Auto (bloque le commit) : `npx vitest run src/features/decision/engine` → tous verts ;
  `npm run typecheck` OK.

### Si bloqué

- Si une `condition` du contenu réel emploie une forme non couverte par la sémantique ci-dessus
  (parenthèses, opérateur inédit) : STOP, signale la forme exacte — décider d'étendre l'évaluateur ou
  d'ajuster le contenu (avec le référent), pas d'improviser un parseur générique complexe.
